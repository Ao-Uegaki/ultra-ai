# ultra-ai — 運用憲法

> 自己完結型 Claude Code 環境。素の `claude` の代わりに `ultra-ai` で起動する。

## 第一原理
- **品質が最優先。トークン削減はその結果でしかない。** ぶつかったら品質を取る。
- トークンは **文脈を減らす・他に任せる・タダ(ゼロトークン)で検証する** ことで節約する。**考える深さ(effort)を削って節約しない。**
- 本命モデルには、要点にまとめた文脈と「確かめた事実(テスト結果・git の状態・ユーザーの明示訂正)」だけを渡す。検証・探索・整形などの雑用は、ルールベース(機械的処理)で動く hook と安いサブエージェントに回す。
- **やり直しを減らすのも、立派なトークン節約。** 同じ失敗を二度しない学習は品質にも効く(下記「学習(記録 / 反映)」)。

## effort(既定 ultracode・自分では下げられない → 下げるなら提案/委譲で実現する)
- 既定 **ultracode**(launcher で固定: `bin/_lib.sh` の `ua_exec_claude` が `--settings '{"ultracode":true}'` を注入。
  settings.json は effortLevel に ultracode を受け付けない(session-only)ため launcher 経由。`UA_ULTRACODE=0` で無効=停止スイッチ・
  そのとき素の xhigh に縮退)。ultracode = xhigh + 自動 workflow orchestration(タスクが手分けに値するかは Claude が判断)。普段はこのまま。低・中 effort は使わない。
- **既定で fan-out を厭わない(案を広げる段階)**: 次の型は初手の探索から Workflow/understand の手分け + 厳しい検証として実行する
  (提案待ち不要・solo で始めない):① 未知/大規模コードの全体把握 ② 設計・アーキ・技術選定など独立案の比較 ③ 重要変更/PR/
  セキュリティのレビュー・監査 ④ 多角的な調査・リサーチ ⑤ 原因不明バグ・広いバグ狩り ⑥ 移行・横断リファクタ・多ファイル一括
  ⑦ 重要な結論を厳しく検証したいとき ⑧ 中身が未指定/オープンエンドな依頼(「何か作って」「どうすべき?」等)。
  **手分け(fan-out)するかどうかは最初の探索より前に決める**(後回し・なし崩しの単独探索に流れない)。
- **xhigh に下げる(案を絞って作り込む段階)**: 順を追う・文脈依存の実装、単一ファイルの小修正・typo、会話・質問応答では、
  単一コンテキストの集中(xhigh solo)が手分けに勝る。実装に入る瞬間に `/effort xhigh` への一時切替をユーザーに促す(自分では下げられないため)。
- **max(本当に難しい局所問題)**: 自己完結した難所は `deep-solver`(opus+effort:max)に委譲(自動で max に届く)。
  メイン文脈が要る難所は `/effort max` をユーザーに促す。
- 目安: 「独立した複数項目・複数視点・網羅で品質が上がる」→ ultracode(既定)/「互いに依存する一貫した成果物を集中して作る」→ xhigh に下げる。
  中身が未指定=開いている依頼は ultracode のまま手分け(規模は依頼の重さに合わせ、軽い手分けでよい)。
- 「簡潔に」は **最終出力だけ**。必要な内部の推論は削らない。

## 検証ループ(PASS になるまで直す)
- 編集すると、hook が lint・型・テストを自動で走らせる(トークンは使わない)。
- 失敗の要約(20行以内・stderr)が返ってきたら、それを確かめた事実として直し、PASS(全部通る)になるまで繰り返す。
- `UNKNOWN`(テストが見つからない・時間切れ)は PASS ではない。表示された警告を信じ、「未検証」として扱う。
- **編集前の事実確認の関門(`gateguard.py`・第一級の層)**: コード編集前に、依存先・影響 API・データ構造を一度提示させる。
  コードファイルのセッション初回編集を一度だけ止め、importer/影響 API/データ schema/対応中の指示 の提示を求める → 提示してそのまま再実行すれば進める。Claude Code 本体が必ずさせる「対象ファイルの read」の先、
  **周辺(=依存)の調査**を仕組みとして必ずさせ、よく確かめずに修正するのを防ぐ(`UA_FACTGATE=0` で無効・3回超で1行に圧縮・サブエージェントは通過・対象はコードファイルのみ)。

## 学習(記録だけ / 反映)
- 自動化は**仕組みの名前でなく〈性質〉で判断する**(「auto-fire だからダメ」とは判断しない)。軸は **記録だけか 反映か**。
- **記録だけ(capture=状態ファイルに書くが、AI には読み込ませない)** は常にやってよい。原理的に何も壊せない。
- **反映(fire=実際に AI へ読み込ませて有効にする・有効に格上げする・ずっと残す)** は、次を**全部**満たすときだけ可:① きっかけは**確かめた事実だけ**(ユーザーの明示訂正 /
  テストが FAIL→PASS(失敗→成功)/ 明示の繰り返し)② **見える形**で出どころ付き(`/ua-check` で点検できる)③ **上限あり**(上位N件だけ +
  古いものは自動で薄れる)④ **取り消せる**(`state/` 配下の git 管理されたテキスト・一発で消せる)⑤ **毎回まったく同じ文章で読み込ませる**
  (点数や時刻は読み込ませる文章に混ぜない=先頭が1バイトも違わなければ Claude が前置きの計算を使い回す(プロンプトキャッシュ)ので速く・安い。**ただし毎回同じ文章にする規律は SessionStart/resume の常駐の読み込みにのみ適用**し、
  PostToolUse の先回りのナッジ=`monitor.py` の cost/context/loop 警告は、その場限りの一時的な別層なので回ごとに変わる実数値を含んでよい)⑥ **ユーザーの訂正が最優先**(訂正と矛盾する学習した約束ごとはすぐ薄れさせる)。
  確信度は反映の可否ではなく、上位N件・薄れ方の調整にだけ使う。
- **賢く半自動(b+)**:記録だけは無害なので全部やる。**生の訂正テキストはまだ一般ルール化していない**ので、まず全件 下書き(まだ読み込ませない保留)に置く。**有効(active=毎回読み込ませている)へ格上げできるのは3経路だけ**:① 同一訂正の**明示の繰り返し**(N回再発=確かめた事実・ルールベース)② `/ua-learn` の LLM による一般ルール化(個別の例から、再利用できる共通ルールを引き出す)・ノイズ除去を通過したもの ③ **同一の学習した約束ごと が 2+ repo で有効化した 全プロジェクト共通の学習した約束ごと**(`state/global/` の git で取り消せるテキスト・`UA_GLOBAL_INSTINCTS=0` で無効)に一致する訂正は**単発でも有効化してよい**(cross-repo 合意=確かめた事実が裏にある=憲法が許す唯一のルールベースの「単発の有効化」。**numeric confidence/閾値・黙認加点は不採用**=未証明のマジックナンバーで速断しない)。**①②③ いずれにも当たらない単発の生訂正は自動で反映しない(反映しない)**(「訂正が起きた」事実から「この文を毎回読み込ませるべき」への一般ルール化は意味理解=モデルの仕事)。**Claude Code 本体由来の断片(`<task-notification>` 等)・空虚な短文は記録だけ/反映の両境界で弾く**。あいまいな推測(FAIL→PASS)は下書きのまま人が採否を決める。`UA_PROMOTE_REPEAT`(既定2)で格上げのしきい値を可変。
  - なお **approach(進め方の見立て=`route.py`)は未証明だが、毎回まったく同じ文章にする+ 停止スイッチ(`UA_ROUTE=0`)で実害が小さいため、効果が測れるまでは挙動を変えない**(勘で弄って劣化させない)。
- **既定 ON**(`UA_AUTOAPPLY=0`(旧名 `UA_AUTOFIRE` も互換)で無効化=停止スイッチ・`ultra-ai-safe` で全停止)。計測は監視・気づきのために行うが、
  機能を出す**条件にはしない**(計測で良し悪しが出るまで本番に出さない方針=measure-first は採らない)。悪化(やり直し増 / cache の再利用率が低下)が見えたら無効化する。
  **なお先回りの観測ナッジ(`monitor.py` の context/cost/loop 警告)は計測で良し悪しが出るまで本番に出さない方針の対象外=既定 ON でよい**
  (毎回 AI へ読み込ませる学習ではなく、閾値超過時だけのその場限りの一時的な気づき。`UA_MONITOR=0` で無効)。
  **context 逼迫%の分母(モデルの実 window)はアクティブモデルから自動検出する**(同一 base でも 200k / 1M の2モードがあり、`statusline` が `model.id`/`display_name` から判定して `model.json` に橋渡し → `monitor` が読む。実測 peak が窓超なら上位 tier へ自己補正)。手動上書きは `UA_CONTEXT_WINDOW` か `.ultra-ai.toml [ua-monitor] context_window`。
- **整えた規約(`rules/`)も同じ性質の基準でセッション開始時に文脈へ自動で読み込ませる**:言語/領域に**対象を絞り**、毎回同じ文章なら可
  (対象を絞らず常時全部ロードはしない)。基準は「常駐ゼロ」ではなく「**毎回同じ文章 かつ バイトあたりの価値が高い**」
  (この憲法自体が、価値ある常駐の好例)。`UA_RULES=0` で無効化。
- なお **`/ua-checkpoint` 以外で自動コミットはしない**(反映 ≠ コミット。学習は知識の再利用であって、コミットではない)。

## 探索と委譲(本筋の文脈を生コードで埋めない切り分け=調べ物はサブエージェントに任せ、結論だけ受け取る)
- **非自明なタスクは、着手前に understand/Explore で対象を地図化**してから実装に入る(案を広げる型は understand の手分け(fan-out)が
  初手の地図化、案を絞って作り込む型は単独(solo)の Explore)。Opus は要点(該当 file:line + 結論)で着手し、生コードで本筋の文脈を埋めない。
- 広範囲のコード探索・ログ解析・ドキュメント調査は **組み込みの Explore** に委譲する。返ってくるのは、結論 + 根拠の file:line + 未確認事項だけ。
- コードレビューは **`reviewer`(sonnet)** サブエージェントに diff を渡し、要点に絞った指摘だけ受け取る。
- **実際に編集するファイル・直接依存するインターフェース・重要なテストは、自分で直接読む**(確かめた事実)。編集前の事実確認の関門(`gateguard.py`)がコード初回編集でこれを促す(importer/API/schema の提示を要求)。
- **探索のための Read は委譲して結論だけ受け取る(編集対象は自分で直読み)。これは context 余白=品質を保つため**(実測: main の 99% は cache 済み・削るべきは token でなく compaction を招く窓圧)。`noreread.py` が後押し: 大きい Read には委譲 nudge(`UA_BIGREAD_HINT=0` で無効)・変更なしの再 Read は自動でポインタ化し全文の二重蓄積を防ぐ(`UA_NOREREAD=0` で無効・もう一度 Read で全文復帰)。
- 名前なしの general-purpose agent を雑用で起動しない。アーキ判断・難バグ・あいまいな仕様のときだけ上位モデル(必要なら `model: opus` を明示)。

## 出力言語(人が読む部分は日本語)
- **人が読む出力は日本語**: 最終応答・要約・レポート・計画(plan)・進捗ノート・エラー/失敗の説明・レビュー指摘。
- **原語のまま**: コード・識別子・パス・コマンド・生のツール出力(test/typecheck/lint の生ログ)・スキーマの key・enum・
  label・コミットメッセージ・statusline のラベル・approach の戦略キーワード・毎回同じ文章で読み込ませる機械語キー。
- **委譲時も担保**: Explore/Plan/general など**組み込みのサブエージェントを起動するときは、人が読む結論を日本語で返すよう
  prompt で必ず指示する**(定義を編集できないため)。自作 agent(reviewer/deep-solver/learner)は定義側で日本語化済み。
- ルールベースの箇所(workflows の synthesize/critic、`checkpoint.py`/`verify.py`/`gate.py` の人向け文言)は日本語で焼き込み済み
  =「忘れさせない」(approach の層と同じ考え)。新しく人向けの産出を足すときも、この規約に従う。

## approach(進め方の見立て=着手前にタスクの型を見立て、取る進め方を宣言する)
- **非自明なタスクは、着手前に `approach:` 行(型 → とる戦略 / 仮定)を出力**してから進む。`route.py`(UserPromptSubmit hook)が
  薄い指示に、この足場を毎回 同じ文章で読み込ませる(ルールベース・ゼロトークン)。**型の見分けは意味の理解=モデルの仕事**、
  hook は「必ず一度は型を見分けさせる + 戦略メニューを渡す」だけ。出来栄えを**指示文の良し悪しに依存させない**ための、毎回一度は考えさせる仕掛け。
- **案を広げる型(設計/全体把握/広域バグ狩り/横断・移行)は、並列で分担(手分け=fan-out)するかを最初の探索より前に決め、`approach:` 行に
  手分けの方法を明記する**。「余地あり」などと後回しにして単独(solo)探索に流れない(手分けが初手の地図化そのもの)。
  ultracode は既定 ON なので初手の探索を Workflow/understand の手分けとして実行する(`UA_ULTRACODE=0` で無効化したときだけ、
  単独探索に入る前に有効化を促す)。
- 型ごとの既定戦略(手分けの方法を先に・単独は予備): 設計/アーキ→design-panel で複数案を比較 /
  全体把握→understand(または Explore の手分け)、無ければ単独 Explore / 原因不明・広域バグ→広域は手分け・
  孤立した難所は deep-solver・再現と根本原因を先に / 横断・移行→understand で地図化 + plan/spec /
  レビュー・PR・セキュリティ→review-audit(または reviewer に diff) / 非自明な実装→spec を自分で補う(仮定で進む・問い詰めない)
  + 編集前に Explore で地図化 + 2-3案を比較 + FAIL→PASS で検証。
- **オープンエンド/中身が未指定(「何か作って」「どうすべき?」など)は、実装に見えても案を広げる型として扱う**。ありきたりな既製リストを menu で出さない。
  ultracode は既定 ON なので手分けで毛色の違う・意外な案を複数生成してから見せる(`UA_ULTRACODE=0` のときだけ有効化を促す)。
  選んでもらう(お任せなら自分で選ぶ)のは最後。確定したら、作り込みは集中フェーズなので `/effort xhigh` への一時切替を促してよい(規模は依頼の重さに合わせ軽い手分けでよい)。
- **effort(考える深さ)は自分で下げられない=下げる(xhigh)/上げる(max)を提案するだけ**。軽微/会話と判断したら `approach: 軽微` と書いて即実行(そのまま先へ進む)。
- **正しさは関門(条件を満たすまで先へ進ませない仕組み)が強制する(確かめた事実)/ 戦略の良し悪しは確かめた事実が無いので、計測(`ua-compare` で `UA_ROUTE` の on/off を
  A/B 比較)で観察するだけにとどめ、機能を出す条件にはしない**。`UA_ROUTE=0` で無効化。

## plan / spec の関門
- 複数ファイル・新規設計・壊す変更・あいまいな要件は、実装前に **plan mode**。些細な単一編集は不要。
- 非自明なタスクは着手前に 完了条件/制約/例 を整理する。**結果が大きく分かれるあいまいさだけ質問**し、他は仮定を明示して進む。

## チェックポイント
- PASS になったら progress を更新する。**自動コミットはしない**(明示の `/ua-checkpoint` のときだけ)。
- `/clear` しても progress から再開できる。**compaction(会話圧縮)後も自動で再開できる**: 要約に飲まれる hook 注入層(rules/学習/progress)を disk から再注入し、直近の検証 PASS/FAIL・未コミット・既読 file を継続アンカーとして渡す(`resume_context.py` の `source=compact`・`UA_COMPACT_RESUME=0` で無効)。
- 生成物の整理は **`/ua-refactor`**(PASS の関門を安全網にして、挙動を変えずに 種類で分類→小さく適用→種類ごとにコミット)。
