現在の作業を、意味あるメッセージで安全にチェックポイント・コミットする。

手順:
1. **メッセージを起案する**: `git diff`(と必要なら `git status`)で追跡済みの変更内容を読み、
   **Conventional Commits 形式の1行**(`<type>(<scope>): <要約>`、type は feat/fix/docs/refactor/test/chore 等)を考える。
   要約は「何を・なぜ」が伝わる簡潔なもの。
2. Bash で `python3 "$CLAUDE_CONFIG_DIR/hooks/checkpoint.py" --message "<起案したメッセージ>"` を実行する。
   - 追跡済みの変更だけを `git add -u` でステージしてコミットする(未追跡ファイルは含めない=機密混入を防ぐ)。
     現ブランチにコミットし、末尾に Co-Authored-By trailer が付く。`--message` 省略時は決定論的な `checkpoint: <files>` へフォールバック。
3. **PASS ゲート**(既定 ON): 直近の検証が FAIL なら拒否される。原則は**直して PASS にしてから**再実行する。
   途中保存がどうしても必要なときだけ `--allow-fail`。検証記録が無い/未検証なら拒否される — 検証を走らせて PASS にするか、
   意図的に未検証保存するなら `--allow-unverified`。**勝手に override を付けず**、FAIL/未検証はまずユーザーに状況を伝える。
4. 新規ファイルも含めたい場合は、**機密でないことを確認**してから明示的に `git add <path>` し、再実行する。
5. 結果(メッセージ・対象ファイル数・ブランチ・コミットハッシュ)を簡潔に報告する。`main`/`master` への直コミット警告が
   出たら伝える。公開したい変更なら、続けて `/ua-ship`(feature ブランチへ push + PR)を案内する。

注意: コミット失敗(git エラー)は検証失敗とは別物。失敗時はそのまま報告し、PASS になるまで直す繰り返しとは混同しない。
